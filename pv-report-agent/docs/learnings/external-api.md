# 외부 API — 공공데이터포털 · nedrug

`DrugPrdtPrmsnInfoService07` 와 nedrug 병행 조회에서 부딪힌 것들.

---

## 1) `DrugPrdtPrmsnInfoService07` 응답 필드 함정

- **상황**: 허가번호를 뽑으려고 `PRDUCT_PRMISN_NO` 필드를 찾음.
- **깨진 것**: 해당 필드 존재하지 않음. 응답에 없는 키를 참조하며 빈 문자열이 채워짐.
- **원인**: 공공데이터포털 스키마에서 허가번호 역할은 **`ITEM_SEQ`** 가 수행. 별도 `PRDUCT_PRMISN_NO` 는 없다. `PERMIT_KIND_CODE` 는 허가·신고 *구분자* 라 혼동 금지.
- **해결**: `lookup_product_info()` 에서 `ITEM_SEQ` 를 허가번호로 사용. `src/product_scraper.py` 의 응답 매핑 주석에 이 근거 남김.
- **재발 방지**: 공공 API 스키마 이름은 직관과 반대. **실제 응답 JSON 을 한 번 덤프** 뜨고 필드 매핑하는 습관.

## 2) `MAIN_ITEM_INGR` 비었을 때 `MATERIAL_NAME` 폴백

- **상황**: 성분명을 `MAIN_ITEM_INGR` 한 필드에서만 읽음.
- **깨진 것**: 일부 품목에서 이 필드가 비어 UI 에서 "성분명 미확인" 으로 표기.
- **원인**: 공공데이터포털이 품목별로 성분 정보를 **`MAIN_ITEM_INGR` 또는 `MATERIAL_NAME`** 중 하나에만 채우는 경우가 있음 (일관성 없음).
- **해결**: `MAIN_ITEM_INGR` 우선, 비면 `MATERIAL_NAME` fallback. 둘 다 비면 빈 문자열.
- **재발 방지**: 공공 API 에서 "주요 필드" 라고 생각한 것이 비어있을 때는 **유사 필드 1~2개 폴백** 을 항상 고려.

## 3) 제품명 검색 → item_seq 재조회 2단 구조

- **상황**: 제품명 검색 한 번으로 상세·주성분까지 얻으려 함.
- **깨진 것**: 검색 응답은 **summary 수준** 이라 성분명·허가번호가 자주 비어있음.
- **원인**: 검색 endpoint 와 상세 endpoint 가 분리되어 있고, 검색은 `item_seq`+요약만 돌려준다.
- **해결**: 사용자가 검색 결과에서 항목 선택 → 선택된 `item_seq` 로 `lookup_product_info` **재호출** 하여 상세·주성분 API enrich.
- **재발 방지**: 공공 API 에서 "검색" 과 "상세" 는 다른 것. 검색 응답만으로 UI 채우지 말 것.

## 4) 403/429/5xx 구분해서 UX 분기

- **상황**: API 실패를 `try/except: st.error("API 오류")` 로 뭉뚱그림.
- **깨진 것**: 사용자가 **만료된 키** 문제인지, **쿼터 초과** 인지, **서버 일시 장애** 인지 구분 못 함 → 무작정 재시도.
- **원인**: 공공데이터포털은 만료키를 403, 쿼터 초과를 429, 서버 장애를 5xx 로 구분해 준다. 이걸 버리면 안 된다.
- **해결**: `lookup_product_info()` 에서 status code 별 사용자 메시지 분기 (`253393e` 커밋).
- **재발 방지**: 외부 API 호출부 예외 처리는 **최소 3분기**(인증 / 쿼터 / 일시장애) 로 시작.
